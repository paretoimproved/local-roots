## Context

Local Roots is a local pickup marketplace with Phases 1-8 complete (v0.8.0.0). All Must Have and Should Have features are built. The goal is a soft launch in Eugene, OR with light marketing, targeting 5-15 farmers from the Lane County Farmers Market scene.

### Implementation Approach
**Approach B: Launch with Safety Net** — Build critical missing pieces, then launch with confidence. Estimated CC effort: ~12-16 hours total (see breakdown table; B4 is highest-risk estimate).

### Market Context
- Eugene, OR: ~175K population, 100+ vendors at Lane County Farmers Market, 24+ CSAs in Lane County, 75+ farms
- Strong local food culture, high demand (produce "always sold out")
- Competitors: Local Line ($49+/mo), Barn2Door ($99-299/mo), GrazeCart ($124+/mo). Local Roots' zero-seller-fee model is genuinely differentiated.
- No existing competitor offers a QR-to-subscription bridge for farmstand walk-ups.

## Vision

### 10x Check
The 10x launch isn't more features — it's a structured launch *program*. Partner with Lane County Farmers Market directly, create a "founding farmer" program with white-glove onboarding, run 4-week structured rollout with weekly farmer cohorts, pre-build Eugene-specific content, and create co-branded QR poster kits. Transform "put it out there" into "community-embedded launch."

## Accepted Scope (Safety Net + Expansions)

### Safety Net Baseline

**B1: FAQ/Help page (~30 min CC)**
- Footer already has `mailto:hello@localroots.com` contact link
- Create `/help` or `/faq` page with common buyer/seller questions
- Link from footer, policies page, and error states

**B2: Seller-initiated refund flow (~3-4 hours CC)**
Sub-tasks:
1. Add `Refund(ctx, paymentIntentID, amountCents) error` method to Stripe client (`stripepay.go`)
2. New backend endpoint: `POST /v1/seller/stores/{storeId}/orders/{orderId}/refund`
3. Handle `charge.refunded` webhook event in `stripe_webhook.go` — register `charge.refunded` in Stripe Dashboard webhook endpoint config (currently only `payment_intent.*` and `account.updated` events are registered)
4. Update `validPaymentTransition` map in `stripe_webhook.go` to allow `paid -> refunded` transition
5. Frontend: "Refund" button on seller order detail for captured/paid orders
Constraints: Full refund only (no partial). Only for orders where `payment_status = 'paid'` AND `status IN ('picked_up', 'ready')`. Note: `paid` is a `payment_status` column value, not an order `status` value — these are separate columns.

**B3: Minimal admin dashboard (~2-3 hours CC)**
- New backend `RequireAdmin` middleware (checks JWT role = "admin"). Consolidate existing inline `role == "admin"` checks in `store_middleware.go`, `seller.go`, and `public.go` into this middleware.
- New admin API endpoints: `GET /v1/admin/dashboard` (aggregated metrics)
- Frontend `/admin` route with client-side role guard (convenience, not security — backend enforces)
- Metrics: active stores count, total subscribers, recent orders, pickup completion rate, revenue

**B4: Refresh token rotation (~4-6 hours CC) ⚠️ HIGHEST-RISK ITEM**
- Current state: JWT TTL is 4 hours across all issuance paths (`auth.go:100`, `buyer_auth.go:172`, `oauth.go:196`). No refresh mechanism.
- Design: issue refresh token (opaque, stored in DB) alongside JWT. Refresh token lifetime: 30 days, single-use rotation. New endpoint: `POST /v1/auth/refresh`.
- Frontend: intercept 401, attempt one refresh. If refresh fails, redirect to login. No request queuing in v1 — keep simple. Must exclude auth pages from 401 interceptor (documented pitfall in CLAUDE.md Common Mistakes).
- Migration: `CREATE TABLE refresh_tokens (id uuid, user_id uuid, token_hash text, expires_at timestamptz, used boolean)`.
- Risk note: This is the most complex item. Backend (token issuance, storage, rotation, revocation) + frontend (401 intercept, token refresh, storage) spans the full stack. Budget extra time.

**B5: Email unsubscribe links (~1 hour CC)**
- Migration: add `email_marketing_opt_out boolean DEFAULT false` to `users` table
- Add unsubscribe link to these marketing templates: `MilestoneCelebration`, `LapsedSubscriberNudge`, `WaitlistNotification`
- Exclude (transactional/operational): `PickupReminder`, `OrderConfirmation`, `SubscriptionConfirmation`, `PostPickupReviewPrompt`, `SellerWeeklyDigest`
- New endpoint: `GET /v1/unsubscribe?token=...` (signed token, sets opt-out flag)
- All marketing email send paths check `email_marketing_opt_out` flag before sending

## Architecture Decisions
- Refund flow: seller-initiated only (no buyer self-service). Seller clicks "Refund" on order → backend calls Stripe Refund API → webhook syncs `payment_status` to `refunded`.
- Admin dashboard: read-only. No admin actions beyond viewing. Backend `RequireAdmin` middleware enforces access. Frontend route guard is convenience only.
- Refresh tokens: opaque tokens stored hashed in DB. Single-use rotation (each refresh invalidates the old token and issues new pair). 30-day lifetime. Revocation via DB delete.
- Email unsubscribe: signed token in URL (no auth required to unsubscribe). Per-user boolean flag. Marketing templates check flag before sending.
- Error pages: Next.js App Router `not-found.tsx` pattern. Global `error.tsx` already exists.
